# SOC 2 — Service Organization Control 2

**Jurisdiction:** USA (globally recognised)  
**Applies to:** Technology and cloud service companies that store, process, or transmit customer data  
**Administered by:** AICPA (American Institute of Certified Public Accountants)  
**Tags:** #compliance #soc2 #security #audit #certification #saas

---

## What It Is

SOC 2 is an auditing standard — not a law. It is a report issued by an independent CPA firm that attests whether a service organisation's controls meet the Trust Services Criteria (TSC). Customers (especially enterprises) frequently require a SOC 2 report before signing contracts.

There are two types:
- **SOC 2 Type I** — Controls are suitably designed at a point in time
- **SOC 2 Type II** — Controls operated effectively over a period (usually 6–12 months) — much stronger and more trusted

---

## The 5 Trust Services Criteria (TSC)

| Category | What it covers | Always required? |
|:---------|:--------------|:----------------|
| **Security** | Protection against unauthorised access | Yes — mandatory |
| **Availability** | System is available as committed | Optional |
| **Processing Integrity** | Processing is complete, valid, accurate | Optional |
| **Confidentiality** | Confidential information is protected | Optional |
| **Privacy** | Personal information is handled per commitments | Optional |

Most companies pursue Security + Availability + Confidentiality as a minimum set.

## Common Controls

### Security (CC6 — Logical and Physical Access)
- Multi-factor authentication on all production systems
- Role-based access control — least privilege
- Offboarding process — access revoked within 24 hours
- Encryption at rest and in transit
- Vulnerability management — regular scans and patching

### Change Management (CC8)
- Code review required before merge
- Automated testing in CI/CD pipeline
- Separate development, staging, and production environments
- Change approval process

### Risk Management (CC3)
- Annual risk assessment
- Documented risk register
- Vendor risk assessments

### Incident Response (CC7)
- Documented incident response plan
- On-call rotation
- Post-incident review process
- Customer notification procedure for breaches

### Monitoring (CC7)
- Centralised logging (SIEM)
- Alerting on anomalous activity
- Regular log review

## The Audit Process

```
1. Readiness Assessment (internal)
   └── Gap analysis against TSC criteria
   └── Implement missing controls

2. Evidence Collection
   └── Screenshots, configurations, policies, logs
   └── User access reviews, vendor assessments

3. Auditor Engagement
   └── Select a licensed CPA firm
   └── Provide evidence for Type I (point in time) or
       evidence over 6–12 months for Type II

4. Audit Report Issued
   └── "Unqualified" = passed
   └── "Qualified" = exceptions noted

5. Annual Renewal
   └── Type II requires annual re-audit
```

## Timeline and Cost

| Phase | Duration | Cost (rough) |
|:------|:---------|:-------------|
| Readiness and control implementation | 3–6 months | $50K–$200K internal effort |
| Type I audit | 1–2 months | $15K–$40K auditor fees |
| Type II audit (12-month period) | Ongoing + 1–2 months | $30K–$80K auditor fees |
| Annual renewal | Ongoing | Similar to above |

## What to Build

```
Minimum SOC 2 Security TSC implementation:

□ MFA on all production and admin systems
□ RBAC — least privilege, access reviews quarterly
□ Offboarding checklist — access revoked within 24h
□ Encryption at rest (AES-256) and in transit (TLS 1.2+)
□ Vulnerability scanning — at least monthly
□ Patch management policy — critical: 7 days, high: 30 days
□ Centralized logging — 90-day retention minimum
□ Incident response plan — documented and tested
□ Change management — code review + approval process
□ Annual risk assessment
□ Security awareness training — annual, all staff
□ Vendor risk assessments for critical vendors
□ Business continuity plan
□ System monitoring and alerting
```
