**Project:** [Platform] (e-commerce marketplace)

> Most teams don't fail on talent — they fail on **friction**: the same problems recur, decisions get re-argued, knowledge lives in one person's head, and blame replaces learning. This doc defines a simple loop so the team improves how it works, every week. ## 📌 The Loop: Pain → Reflection → Document → Improve → Trust

```
   +------------------+      +------------------+      +------------------+
   |  1 · PAIN        | ===> |  2 · REFLECTION  | ===> |  3 · DOCUMENT    |
   |  notice friction |      |  ask WHY (not    |      |  write the lesson|
   |                  |      |  WHO)            |      |  + the decision  |
   +------------------+      +------------------+      +------------------+
            ^                                                   |
            |                                                   v
   +------------------+                              +------------------+
   |  5 · TRUST       | <=== run a set duration <=== |  4 · IMPROVE     |
   |  the system works|      then review again       |  change the flow |
   +------------------+                              +------------------+

   Rule: when something breaks -> ask WHY & fix the process.
         Don't blame a person ("don't kill the goat").
```

| Step | What it means | For this team |
|:-----|:--------------|:--------------|
| **1 · Pain** 😖 | Notice the friction — what slowed us down or went wrong | "We re-argued the platform choice 3 times" |
| **2 · Reflection** 🪞 | Ask **why** it happened — the process, not the person | "Because we never wrote the decision down" |
| **3 · Document** 📄 | Write the lesson + the decision where everyone can find it | Record it in [tech-stack.md](./tech-stack.md) |
| **4 · Improve** 🔧 | Change the way we work so it can't recur | "All decisions go in the relevant doc, same day" |
| **5 · Trust** 🤝 | The fix holds; the team trusts the system → repeat the loop | Fewer repeated arguments next sprint |

## The 3 rules that prevent most issues

### 1. Decisions get written down, the same day
The #1 source of team friction is **re-deciding things**. If it's not written in a doc ([tech-stack](./tech-stack.md), [business-model-canvas](./business-model-canvas.md), [decisions-needed](./decisions-needed.md)), it didn't happen. *"Where did we land on X?"* should always have a link as the answer.

### 2. Blameless problem-solving — ask *why*, not *who* 🐐
When something breaks, the team asks **"why did our process allow this?"** and fixes the process — never hunts for someone to blame ("don't kill the goat"). Blame makes people hide problems; safety makes them surface problems early, while they're still cheap to fix.

### 3. One clear owner per area + per decision
Every responsibility area ([responsibilities.md](./responsibilities.md)) and every open decision ([decisions-needed.md](./decisions-needed.md)) has **one owner**. Shared ownership = no ownership. The owner drives it; the team supports.

## The weekly rhythm (the loop in practice)

Run the loop on a fixed cadence — the [weekly meeting](./weekly-meeting.md):

1. **Review** — what did we commit to last week? What actually happened? *(Reflection)*
2. **Surface pains** — what caused friction this week? *(Pain — say it openly, no blame)*
3. **Decide & assign** — make the open decisions; give each an owner + date *(Improve)*
4. **Document** — write down every decision before leaving the meeting *(Document)*
5. **Commit** — agree next week's focus *(Trust the plan)*

> Keep it short. Every meeting should end with **written decisions and clear owners** — not just discussion.

## Common pains for this team → how the loop handles each

| Pain (likely) | Loop response |
|:--------------|:--------------|
| Same decision re-argued | Document it once → link to it next time |
| "I thought *you* were doing that" | One owner per area ([responsibilities.md](./responsibilities.md)) |
| A teammate's mistake breaks something | Ask why the process allowed it; fix the process, not the person |
| Knowledge stuck with one person | Write it in a shared doc → anyone can pick it up |
| Meetings with no outcome | Every meeting ends in written decisions + owners |
| Stalled because a decision is unmade | Track it in [decisions-needed.md](./decisions-needed.md) with an owner + date |

---

## The one principle

> **A team collaborates well not because no problems happen — but because every problem becomes a documented improvement instead of a recurring argument.** Stay curious, stay humble, keep the loop turning.
