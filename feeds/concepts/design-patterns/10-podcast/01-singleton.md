## ១. កិច្ចសន្ទនា (Podcast Transcript)

**Host A (សិទ្ធិ - អ្នកជំនាញ):** សួស្តីអ្នកទាំងអស់គ្នា! ស្វាគមន៍មកកាន់កម្មវិធី Engineering Podcast របស់យើង។ ថ្ងៃនេះយើងនឹងជជែកគ្នាអំពី **Singleton Pattern**។ លោក បូរិទ្ធ តើលោកធ្លាប់មានបញ្ហាដែលកម្មវិធីដំណើរការយឺត ដោយសារតែបង្កើត Object Database Connection ច្រើនពេកទេ?

**Host A (សិទ្ធិ):** ហាហា! ហ្នឹងហើយគឺជាបញ្ហា! ដើម្បីដោះស្រាយរឿងនេះ យើងប្រើ Singleton។ ស្រមៃមើលធនាគារមួយ។ ធនាគារមានបុគ្គលិកបញ្ជររាប់សិបនាក់ ប៉ុន្តែពួកគេមាន "ទូដែកធំ" តែមួយគត់។ គ្រប់គ្នាមិនមែនមានទូដែកផ្ទាល់ខ្លួនឯងនៅក្រោយតុនោះទេ បើមិនអញ្ចឹងទេ គ្មាននរណាដឹងថាធនាគារមានលុយពិតប្រាកដប៉ុន្មានឡើយ។ ដូច្នេះទូដែកនោះគឺជា Single Source of Truth។

**Host B (បូរិទ្ធ):** ស្តាប់ទៅឡូយណាស់! ចុះ... សុំសួរឆ្កួតៗមួយបង! ចុះបើខ្ញុំជាបុគ្គលិកខិលខូច ខ្ញុំលួច Copy សោរទូដែកហ្នឹង ទៅវាយធ្វើទូដែកក្លែងក្លាយមួយទៀតនៅផ្ទះ តើប្រព័ន្ធបងមានវិធីការពារម៉េចទៅ?

**Host A (សិទ្ធិ):** សំណួរឆ្កួតមែន តែឆ្លាត! នៅក្នុងកូដ ការ "វាយទូដែកថ្មី" គឺស្មើនឹងការហៅពាក្យ `new`។ ដើម្បីការពារអ្នកខិលខូចដូចលោកបូរិទ្ធ យើងត្រូវចាក់សោ `constructor` របស់ Class នោះឱ្យទៅជា `private`។ នៅពេល `constructor` ត្រូវបិទ ទោះលោកឯងខំវាយ `new DatabasePool()` រាប់ពាន់ដងទៀត ក៏ Compiler វៃក្បាលលោកឯងវិញដែរ!

**Host A (សិទ្ធិ):** កុំភ័យ! យើងមានច្រកទ្វារ V.I.P មួយ គឺ Method `public static getInstance()`។ ពេលលោកឯងហៅវា វានឹងឆែកមើល៖ បើទូដែកមិនទាន់មាន វានឹងហៅជាងមកសង់ទូដែកថ្មីមួយ។ តែបើមានរួចហើយ វាគ្រាន់តែចង្អុលបង្ហាញផ្លូវឱ្យលោកឯងដើរទៅប្រើទូដែកចាស់នោះជាការស្រេច។

**Host B (បូរិទ្ធ):** ហាហា... ចុះបើខ្ញុំហៅមិត្តភក្តិ ១០០ នាក់ (Threads) ឱ្យរត់ចូលទៅច្រកទ្វារ V.I.P ហ្នឹងព្រមគ្នាក្នុងវិនាទីតែមួយ (Race Condition) ម៉េចដែរ? ជាងសង់ទូដែកច្បាស់ជាវង្វេង សង់ទូដែក ១០០ អោយពួកខ្ញុំមិនខាន!

**Host A (សិទ្ធិ):** នែ៎! នៅមានល្បិចទៀត! ហ្នឹងហើយជាកន្លែងដែល Singleton អាចបែកធ្លាយពិតមែន។ ដើម្បីទប់ទល់នឹងក្បាច់ហៅគ្នាមកសម្រុកនេះ នៅក្នុង Java យើងប្រើតិចនិក "Double-Checked Locking"។ វាដូចជាមាន Security Guard ពីរជាន់អញ្ចឹង។ ជាន់ទី១ ឆែកមើលក្រៅទ្វារ ជាន់ទី២ ចាក់សោទ្វារ (synchronized) រួចឆែកម្តងទៀត។ ធានាថាមក ១០០០ នាក់ព្រមគ្នាក៏ចូលបានតែម្តងម្នាក់ដែរ ហើយទូដែកតែមួយគត់នឹងត្រូវបានសង់ឡើងយ៉ាងរឹងមាំ!

## ២. ការបកប្រែទៅជាកូដ (Translating to Code)

```java
public class DatabasePool {
    // 1. volatile ការពារ Security Guard កុំឱ្យវង្វេង (ការពារ Reordering)
    private static volatile DatabasePool instance;

    // 2. private constructor ការពារបូរិទ្ធមិនឱ្យប្រើ new
    private DatabasePool() { }

    // 3. Double-Checked Locking សម្រាប់ទាញយក Object
    public static DatabasePool getInstance() {
        if (instance == null) { // ឆែកជាន់ទី១
            synchronized (DatabasePool.class) { // ចាក់សោទ្វារ
                if (instance == null) { // ឆែកជាន់ទី២ ការពារហ្វូង Thread
                    instance = new DatabasePool();
                }
            }
        }
        return instance;
    }
}
```

## ៣. គុណសម្បត្តិ និងគុណវិបត្តិ (Pros & Cons)

**គុណសម្បត្តិ (Pros):**
- **ការពារការបំផ្លាញម៉ាស៊ីន:** ការពារការបង្កើត Object ដែលប្រើ Memory ច្រើនដដែលៗ។
- **Single Source of Truth:** ធានាថាកម្មវិធីទាំងមូលប្រើប្រាស់ទីតាំងតែមួយគត់។

**គុណវិបត្តិ (Cons):**
- **Global State អាក្រក់ជូរជាតិ:** ពេលធ្វើ Unit Test ទូដែកនេះវានៅជាប់រហូត ធ្វើឱ្យ Test មួយមានឥទ្ធិពលលើ Test មួយទៀត ដូចខ្មោចលងអញ្ចឹង។
- **សត្រូវរបស់ Microservices:** បើ Server អ្នកមាន ៣ គ្រឿង អ្នកនឹងមាន Singleton ៣ មិនមែនមួយទេ!

## ៤. ចំណុចសំខាន់ៗ (Key Takeaways)

- **បញ្ហា:** កម្មវិធីត្រូវការវត្ថុតែមួយគត់ ដើម្បីចៀសវាងការខ្ជះខ្ជាយ។
- **ដំណោះស្រាយ:** បិទ `constructor` ឱ្យទៅជា `private` និងប្រើ `getInstance()` ជាមួយ Double-Checked Locking។
