## ១. ទិដ្ឋភាពក្នុងថ្នាក់រៀន (Classroom Dialogue)

*(អាម៉ាប់កំពុងសរសេរកម្មវិធីគ្រប់គ្រងសត្វក្នុងសួនសត្វ ដែលមានសត្វតោ ដំរី និងស្វា។ លោកគ្រូអោយអាម៉ាប់ថែមមុខងារ "ចាក់វ៉ាក់សាំង" អោយសត្វទាំងអស់។ អាម៉ាប់ក៏រត់ទៅបើក Class តោ, ដំរី, ស្វា មួយៗ ដើម្បីសរសេរកូដ `vaccinate()` បញ្ចូល)*

**លោកគ្រូយក្ស:** អាម៉ាប់! ហែងកំពុងធ្វើស្អី? ចុះបើថ្ងៃស្អែកអញអោយថែមមុខងារ "ថ្លឹងទម្ងន់", "កាត់ក្រចក", "បកចៃ" ទៀត ហែងមិនរត់ចូលទៅកែ Class សត្វទាំងនោះវឹកវរពេញសួនសត្វទេអី? កូដប្រឡាក់ប្រឡូសអស់ហើយ!

**អាម៉ាប់:** លោកគ្រូ! ក៏មុខងារហ្នឹងធ្វើលើសត្វ អញ្ចឹងខ្ញុំត្រូវតែចូលទៅកែក្នុង Class សត្វហ្នឹងហើយ! បើមិនអញ្ចឹងអោយខ្ញុំសរសេរនៅឯណា? នៅលើមេឃមែន?

**លោកគ្រូយក្ស (គប់ហ្វឺត):** អាភ្លើ! ហ្នឹងគេហៅថាបំពាន OCP! គេត្រូវប្រើ **Visitor Pattern** វើយ! 
ហែងត្រូវចាត់ទុកមុខងារ "ចាក់វ៉ាក់សាំង" ហ្នឹងថាជា "ភ្ញៀវ (ពេទ្យសត្វ)"! ពេទ្យសត្វនេះគាត់ដើរទៅជួបសត្វម្នាក់ម្តងៗ ដើម្បីចាក់ថ្នាំ។ នៅក្នុង Class សត្វ ហែងគ្រាន់តែបើក "ទ្វារស្វាគមន៍ (`accept()`)" អោយពេទ្យចូលទៅបានហើយ! ពេទ្យជាអ្នកកាន់កូដចាក់វ៉ាក់សាំង (Logic) ដោយខ្លួនឯង! សត្វមិនបាច់ចេះចាក់វ៉ាក់សាំងខ្លួនឯងទេ!

**អាម៉ាប់ (អេះក្បាល):** អូហូ! មានពេទ្យដើរចាក់ដល់ផ្ទះ! ចុះសុំសួរមួយគ្រូ! ពេលពេទ្យ (Visitor) ហ្នឹងដើរទៅជួប "តោ" តើពេទ្យគាត់ប្រើថ្នាំដំរី ឬថ្នាំតោទៅចាក់វាលោកគ្រូ? ម៉េចពេទ្យដឹងថាគាត់កំពុងជួបអ្នកណា? 

**លោកគ្រូយក្ស (សម្លក់):** នេះហើយជាភាពអស្ចារ្យដែលហៅថា "Double Dispatch" អាម៉ាប់!
ពេលពេទ្យដើរចូលទ្វារស្វាគមន៍របស់តោ (`accept()`) តោនឹងស្រែកប្រាប់ពេទ្យថា "លោកពេទ្យ ខ្ញុំជាតោ! `visitor.visit(this)`!" ពេលនោះ ពេទ្យនឹងទាញយកកូដ Method សម្រាប់ព្យាបាលតោ (`visit(Lion)`) មកប្រើដោយស្វ័យប្រវត្តិ! អត់មានច្រឡំយកថ្នាំស្វាទៅចាក់តោទេវើយ!

**អាម៉ាប់:** អូ! តោជាអ្នកស្រែកប្រាប់! បែបនេះស្រួលណាស់! បើចង់ថែមមុខងារ "បកចៃ" ខ្ញុំគ្រាន់តែបង្កើត Class ពេទ្យបកចៃមួយទៀត រួចអោយគាត់ដើរចូលទ្វារស្វាគមន៍របស់សត្វដដែលហ្នឹងទៅ វាចប់បាត់ហើយ! មិនបាច់កែកូដសត្វទៀតទេ!

## ២. កូដនៅលើក្តារខៀន (Code on the Blackboard)

```java
// ១. ភ្ញៀវ (ពេទ្យសត្វ ដែលផ្ទុកមុខងារថ្មីៗ)
interface AnimalVisitor {
    void visit(Lion lion);     // វិធីព្យាបាលតោ
    void visit(Monkey monkey); // វិធីព្យាបាលស្វា
}

// ២. ពេទ្យចាក់វ៉ាក់សាំង (មុខងារថ្មីទី១)
class VaccineVisitor implements AnimalVisitor {
    public void visit(Lion lion) { System.out.println("ចាក់ថ្នាំអោយតោដោយប្រុងប្រយ័ត្ន..."); }
    public void visit(Monkey monkey) { System.out.println("លួងស្វាអោយស៊ីចេកសិន ចាំចាក់ថ្នាំ..."); }
}

// ៣. សត្វ (អ្នកផ្តល់ទ្វារស្វាគមន៍ មិនបាច់កែប្រែអ្វីទៀតទេ)
interface Animal {
    void accept(AnimalVisitor visitor);
}

class Lion implements Animal {
    public void accept(AnimalVisitor visitor) {
        // Double Dispatch: ខ្ញុំជាតោ (this)! សូមជួយព្យាបាលខ្ញុំផង!
        visitor.visit(this); 
    }
}

class Monkey implements Animal {
    public void accept(AnimalVisitor visitor) {
        visitor.visit(this);
    }
}
```

## ៣. ពិន័យ និងមេរៀន (Punishment & Lesson Learned)

**ពិន័យសម្រាប់អាម៉ាប់:**
- ត្រូវដើរទៅជួបសិស្សក្នុងថ្នាក់ម្តងម្នាក់ៗ ហើយស្រែកថា "ខ្ញុំជាភ្ញៀវ ខ្ញុំមកលេង!" បើហ៊ានដើរខុសតុ គ្រូនឹងអោយរត់ជុំវិញថ្នាក់!

**មេរៀនដែលទទួលបាន (Lesson Learned):**
- **បញ្ហា:** ការបន្ថែមមុខងារថ្មីៗទៅកាន់ Objects ច្រើនប្រភេទ (រចនាសម្ព័ន្ធស្មុគស្មាញ) តម្រូវអោយយើងចូលទៅកែប្រែ Classes ទាំងនោះញឹកញាប់ ដែលនាំអោយប្រឈមនឹងការខូចកូដចាស់ៗ។
- **ដំណោះស្រាយ:** ប្រើប្រាស់ Visitor Pattern ដោយដកកូដមុខងារថ្មីៗចេញមកដាក់ក្នុង Visitor Class រួចប្រើប្រាស់យន្តការ "Double Dispatch" ដើម្បីអោយ Visitor អាចប្រតិបត្តិការទៅលើ Object នីមួយៗបានយ៉ាងត្រឹមត្រូវ ដោយមិនបាច់កែប្រែរចនាសម្ព័ន្ធរបស់ពួកវាឡើយ។
